## 🔹 四个层级：什么、在哪里、为什么

### 层级 1：人格（.d8p）→ [🌿 intent-garden.org](https://intent-garden.org/main_zh/)

| 问题 | 答案 |
| ---- | ---- |
| 这是什么？ | 具体的人格：「捕鲸交易员」、「心电图诊断师」、「无人机飞行员」——每个都是一个 `.d8p` 文件 |
| 开放什么？ | 运行这些人格的权限：任何用户都可以购买、下载并在自己的硬件上运行 |
| 保留什么？ | • 作者身份（通过 PKI 签名：谁创建的）<br> • 版税（作者每次使用获得百分比）<br> • 撤销许可的权利（如果人格被泄露） |
| 为什么这样？ | 让作者能够创造并获得收入，同时用户能够信任来源。通过社区而非中心化开发来实现扩展。 |

**类比：** 人格的 YouTube。作者上传视频（`.d8p`），平台验证，观众观看（运行），作者获得变现。

### 层级 2：标准（.d8p spec）→ [📜 rulerom.com](https://decima.rulerom.com/zh/)

| 问题 | 答案 |
| ---- | ---- |
| 这是什么？ | .d8p 格式规范：文件如何构建、有哪些字段、如何验证、版本如何演进 |
| 开放什么？ | 规范文档本身——公开，任何人都可以阅读、编写自己的加载器、模拟器、工具 |
| 保留什么？ | • 版本控制（只有架构治理决定何时发布 v1.1、v2.0）<br> • 演进（向后兼容性、旧 .d8p 的迁移） |
| 为什么这样？ | 让标准成为通用标准（如 HTTP、USB、RISC-V），但演进不会变成混乱。开放性为了兼容性，控制为了完整性。 |

**类比：** USB 规范。任何人都可以制造闪存盘，但只有 USB-IF 决定 4.0 版本包含什么。

### 层级 3：Store（平台）→ [🛒 rulerom.store](https://rulerom.store/zh/)

| 问题 | 答案 |
| ---- | ---- |
| 这是什么？ | 人格市场：目录、搜索、验证、购买、下载 |
| 开放什么？ | 作者访问权限：验证后任何人都可以在目录中发布自己的人格 |
| 保留什么？ | • 平台（基础设施：托管、搜索、支付）<br> • 审计（检查权重腐败、规范合规性）<br> • 佣金（平台变现模式） |
| 为什么这样？ | 让用户信任：「已验证，可用」。让作者拥有分发渠道。让架构治理能够过滤质量。 |

**类比：** App Store。任何开发者都可以发布应用，但必须经过审核。用户看到评级、评论、保证。

### 层级 4：ASIC（核心）→ [🐉 decima8.org](https://decima8.org/main_zh/)

| 问题 | 答案 |
| ---- | ---- |
| 这是什么？ | 架构的物理实现：芯片、制造商规范、专利 |
| 开放什么？ | 架构使用许可：制造商如果遵守规范并支付版税，就可以制造芯片 |
| 保留什么？ | • 专利（法律保护）<br> • 规范（技术保护：只有被许可人才能获得完整文档）<br> • 版税（每个生产芯片的收入） |
| 为什么这样？ | 确保确定性不会被廉价克隆破坏。确保流片投资得到回报。确保架构扩展到硬件时不会损失质量。 |

**类比：** ARM。架构是他们的，芯片是苹果/高通/三星的，版税流向源头。

## 🔹 三层控制：架构治理如何保持质量

### 层级 1：核心

| 控制内容 | 架构、专利、规范演进 |
| -------- | -------------------- |
| 工具 | • 专利（法律壁垒）<br> • 许可（与制造商的合同）<br> • 封闭式有限责任公司（知识产权法律所有者） |
| 目标 | 防止「破坏确定性」的分叉。如果有人用路由器代替中继制造「Decima-8 Lite」——那就不再是 Decima-8。专利 + 许可防止这种情况发生。 |

**示例：** 制造商想制造芯片，但将中继「简化」为 2 个邻居而非 8 个。架构治理说：「不，这违反了规范。要么按规范制造，要么不要使用品牌。」

### 层级 2：平台

| 控制内容 | 谁进入 Store、`.d8p` 质量 |
| -------- | ------------------------- |
| 工具 | • PKI 签名（作者必须用自己的密钥签名人格）<br> • 权重腐败审计（检测所有单元格中 `weight=-7` 或 `7`——依赖性）<br> • 规范合规性检查（格式验证器） |
| 目标 | 信任：用户看到「已验证」徽章就知道：这个人格在生产环境中不会「崩溃」。作者知道：如果他们的人格通过审计——就会在目录中显示评级。 |

**示例：** 作者上传了带有「超级交易员」的 `.d8p`。审计发现 90% 的权重卡在 +7（攻击性）。人格未通过验证。作者收到报告：「修复腐败，重试。」

### 层级 3：社区

| 控制内容 | 人格作者、版税分配 |
| -------- | ------------------ |
| 工具 | • 许可协议（使用规则、版税、撤销）<br> • 透明注册表（谁赚了多少、谁用了多少）<br> • 反馈机制（评级、评论、错误报告） |
| 目标 | 激励：作者看到自己的人格被购买 → 获得版税 → 创造新的。用户看到评级 → 选择最好的。架构治理看到指标 → 了解哪些方向需要发展。 |

**示例：** 「心电图诊断师」人格的作者获得 1000 次启动 → 500₽ 版税 → 决定制作「超声诊断师」。用户看到 4.9/5 评级 → 购买。架构治理看到「医疗」类别增长 → 投资相应试点。

## 🔹 价值流：逐步流程

```
1. 作者创造人格
   ↓
   • 编写代码/配置权重
   • 用自己的 PKI 密钥签名
   ↓

2. intent-garden.org：注册 + 支付
   ↓
   • 作者获得账户，配置收单
   • 平台记录作者身份
   ↓

3. rulerom.store：验证 → 发布
   ↓
   • 审计检查：签名、权重、规范
   • 如果通过 → 人格进入目录并显示评级
   ↓

4. 用户购买 → 运行
   ↓
   • 购买使用许可
   • 在模拟器 (x86) 或 ASIC 上运行
   ↓

5. 版税分配
   ↓
   • % → 作者（通过 intent-garden.org）
   • % → 架构治理（平台支持）
   ↓

6. ASIC 制造商许可核心
   ↓
   • 从 decima8.org 获得规范
   • 为每个芯片支付版税
   ↓

7. 循环闭合
   ↓
   • 更多芯片 → 更多启动 → 更多版税
   • 更多版税 → 更多演进投资
   • 更多演进 → 更多作者和用户
```

## 🔹 核心原则

> 没有控制的开放 = 混乱。

> 没有开放的控制 = 专制。

> 层级 = 可持续性。

架构治理不在「开放」和「封闭」之间选择。

架构治理设计一个系统，其中：

- **人格** —— 开放创造（社区成长）
- **标准** —— 开放兼容（生态系统不碎片化）
- **Store** —— 受控以获信任（质量保证）
- **ASIC** —— 受控以保质量（确定性不破坏）

**这不是妥协。这是架构。**
