## 🔹 Четыре слоя: что где и зачем

### Слой 1: Личности (.d8p) → [🌿 intent-garden.org](https://intent-garden.org/main_ru/)

| Вопрос | Ответ |
| ------ | ----- |
| Что это? | Конкретные личности: «трейдер-китобой», «диагност-ЭКГ», «пилот-дрон» — каждая как файл `.d8p` |
| Что отдаётся? | Доступ к запуску этих личностей: любой пользователь может купить, скачать, запустить на своём железе |
| Что сохраняется? | • Авторство (через PKI-подпись: кто создал)<br> • Роялти (автор получает % за каждое использование)<br> • Право отозвать лицензию (если личность скомпрометирована) | 
| Зачем так? | Чтобы авторы могли творить и получать доход, а пользователи — доверять источнику. Масштабирование через сообщество, а не через централизованную разработку. |

**Аналогия:** YouTube для личностей. Автор загружает видео (`.d8p`), платформа проверяет (верификация), зритель смотрит (запускает), автор получает монетизацию.

### Слой 2: Стандарт (.d8p spec) → [📜 rulerom.com](https://decima.rulerom.com/ru/)

| Вопрос | Ответ |
| ------ | ----- |
| Что это? | Спецификация формата .d8p: как устроен файл, какие поля, как валидировать, как эволюционирует версия |
| Что отдаётся? | Сам документ спецификации — открыт, любой может прочитать, написать свой загрузчик, эмулятор, инструмент |
| Что сохраняется? | • Контроль версий (только Управление Архитектуры решает, когда выпускать v1.1, v2.0)<br> • Эволюция (обратная совместимость, миграция старых .d8p) |
| Зачем так? | Чтобы стандарт стал общим (как HTTP, USB, RISC-V), но эволюция не превратилась в хаос. Открытость для совместимости, контроль для целостности. |

**Аналогия:** Спецификация USB. Любой может сделать флешку, но только USB-IF решает, что будет в версии 4.0.

### Слой 3: Store (платформа) → [🛒 rulerom.store](https://rulerom.store/ru/)

| Вопрос | Ответ | 
| ------ | ----- |
| Что это? | Маркетплейс личностей: каталог, поиск, верификация, покупка, загрузка |
| Что отдаётся? | Доступ для авторов: после верификации любой может опубликовать свою личность в каталоге |
| Что сохраняется? | • Платформа (инфраструктура: хостинг, поиск, платежи)<br> • Аудит (проверка на коррупцию весов, соответствие спеке)<br> • Комиссии (модель монетизации платформы) |
| Зачем так? | Чтобы пользователи доверяли: «проверено, работает». Чтобы авторы имели канал дистрибуции. Чтобы Управление Архитектуры могло фильтровать качество. |

**Аналогия:** App Store. Любой разработчик может опубликовать приложение, но только после ревью. Пользователь видит рейтинг, отзывы, гарантию.

### Слой 4: ASIC (ядро) → [🐉 decima8.org](https://decima8.org/main_ru/)

| Вопрос | Ответ |
| ------ | ----- |
| Что это? | Физическая реализация архитектуры: чип, спецификации для производителей, патенты |
| Что отдаётся? | Лицензия на использование архитектуры: производитель может делать чипы, если соблюдает спецификацию и платит роялти |
| Что сохраняется? | • Патенты (юридическая защита)<br> • Спецификации (техническая защита: только лицензиаты получают полные доки)<br> • Роялти (доход за каждый произведённый чип) |
| Зачем так? | Чтобы детерминизм не был скомпрометирован дешёвыми клонами. Чтобы инвестиции в tapeout окупались. Чтобы архитектура масштабировалась в железо без потери качества. |

**Аналогия:** ARM. Архитектура у них, чипы у Apple/Qualcomm/Samsung, роялти течёт к источнику.

## 🔹 Три уровня контроля: как Управление Архитектуры держит качество

### Уровень 1: Ядро

| Что контролируется | Архитектура, патенты, эволюция спецификации |
| ------------------ | ------------------------------------------- |
| Инструмент | • Патенты (юридический барьер)<br> • Лицензирование (договор с производителями)<br> • Закрытое ООО (юридический владелец ИС) |
| Цель | Защита от форков, которые «сломают детерминизм». Если кто-то сделает «Decima-8 Lite» с роутерами вместо эстафеты — это уже не Decima-8. Патенты + лицензии не дают этому случиться. |

**Пример:** Производитель хочет сделать чип, но «упростить» эстафету до 2 соседей вместо 8. Управление Архитектуры говорит: «Нет, это нарушает спецификацию. Либо делайте по спеке, либо не используйте бренд».

### Уровень 2: Платформа

| Что контролируется | Кто попадает в Store, качество `.d8p` |
| ------------------ | ------------------------------------------- |
| Инструмент | • PKI-подпись (автор должен подписать личность своим ключом)<br> • Аудит на коррупцию весов (детекция `weight=-7` или `7` по всем ячейкам — зависимость)<br> • Проверка соответствия спеке (валидатор формата) |
| Цель | Доверие: пользователь видит бейдж «Проверено» и знает: эта личность не «сломается» в продакшене. Автор знает: если его личность пройдёт аудит — она будет в каталоге с рейтингом. |

**Пример:** Автор загрузил `.d8p` с «супер-трейдером». Аудит обнаружил, что 90% весов застряли в +7 (агрессия). Личность не проходит верификацию. Автор получает отчёт: «Исправьте коррупцию, попробуйте снова».

### Уровень 3: Сообщество

| Что контролируется | Авторы личностей, распределение роялти |
| ------------------ | ------------------------------------------- |
| Инструмент | • Лицензионный договор (правила использования, роялти, отзыв)<br> • Прозрачный реестр (кто сколько заработал, кто сколько использовал)<br> • Механизм обратной связи (рейтинги, отзывы, баг-репорты) |
| Цель | Стимул: автор видит, что его личность покупают → получает роялти → создаёт новую. Пользователь видит рейтинг → выбирает лучшее. Управление Архитектуры видит метрики → понимает, какие направления развивать. |

**Пример:** Автор личности «Диагност-ЭКГ» получил 1000 запусков → 500₽ роялти → решил сделать «Диагност-УЗИ». Пользователь видит рейтинг 4.9/5 → покупает. Управление Архитектуры видит рост категории «медицина» → инвестирует в соответствующие пилоты.

## 🔹 Поток ценности: шаг за шагом

```
1. Автор создаёт личность
   ↓
   • Пишет код / настраивает веса
   • Подписывает своим PKI-ключом
   ↓

2. intent-garden.org: регистрация + оплата
   ↓
   • Автор получает аккаунт, эквайринг настроен
   • Платформа фиксирует авторство
   ↓

3. rulerom.store: верификация → публикация
   ↓
   • Аудит проверяет: подпись, веса, спека
   • Если ОК → личность в каталоге с рейтингом
   ↓

4. Пользователь покупает → запускает
   ↓
   • Покупает лицензию на использование
   • Запускает на эмуляторе (x86) или ASIC
   ↓

5. Роялти распределяются
   ↓
   • % → Автору (через intent-garden.org)
   • % → Управлению Архитектуры (поддержка платформы)
   ↓

6. Производитель ASIC лицензирует ядро
   ↓
   • Получает спецификацию с decima8.org
   • Платит роялти за каждый чип
   ↓

7. Цикл замыкается
   ↓
   • Больше чипов → больше запусков → больше роялти
   • Больше роялти → больше инвестиций в эволюцию
   • Больше эволюции → больше авторов и пользователей
```

## 🔹 Главный принцип

> Открытость без контроля = хаос.

> Контроль без открытости = диктатура.

> Слои = устойчивость.

Управление Архитектуры не выбирает между «открыто» и «закрыто».

Управление Архитектуры проектирует систему, где:

- **Личности** — открыты для творчества (сообщество растёт)
- **Стандарт** — открыт для совместимости (экосистема не фрагментируется)
- **Store** — контролируем для доверия (качество гарантировано)
- **ASIC** — контролируем для качества (детерминизм не ломается)

**Это не компромисс. Это — архитектура.**
